OVERVIEW
--------

This repository has two active product tracks:

1) Core Coherence Engine (stable):
   Local-first coherence analysis with layered scoring, composite scoring, and
   explanation outputs across CLI, GUI, and legacy API surfaces.

2) Fund Orchestrator Backend (active buildout):
   API + persistence + workers for intake, scoring orchestration, policy-driven
   decisioning, and outbox-based event dispatch.

Release readiness checklist + full docs sync landed. The 20-prompt

**New: deterministic release readiness checklist.**

``deploy/scripts/release_readiness_check.py`` is a loose, self-pathing
script (it inserts the repository parent on ``sys.path`` the same way
``deploy/scripts/merge_governed_historical_outcomes.py`` does) that
executes ten checks in a stable registered order. Each check returns a
``CheckResult`` with a ``check_id``, ``status`` (``pass`` / ``fail`` /
``error``), a ``reason_code`` drawn from a closed vocabulary, a
human-readable ``detail`` string, and an ``evidence`` mapping. The
registered checks are:

  - ``alembic_head_present`` — reads the on-disk Alembic tree via
    ``alembic.config.Config`` and ``alembic.script.ScriptDirectory``,
    asserts exactly one head revision, and confirms the matching
    migration file exists under ``alembic/versions/``.
  - ``decision_policy_version_pinned`` — imports
    ``server/fund/services/decision_policy.py`` and asserts
    ``DECISION_POLICY_VERSION == "decision-policy-v1"`` (the pin set by
    prompt 01).
  - ``event_schemas_load_and_validate`` — iterates
    ``SUPPORTED_EVENTS`` from prompt 02, loads each schema, pulls the
    first example payload from ``examples[]``, and runs
    ``validate_event`` against it so regressions in schemas or
    examples are caught before merge.
  - ``prompt_registry_verifies`` — loads the SHA-pinned prompt
    registry from prompt 08 via ``load_registry`` and calls
    ``verify_registry``; any drift in ``sha256`` or missing templates
    is surfaced here.
  - ``e2e_integration_test_marked`` — verifies
    ``tests/integration/test_e2e_pipeline.py`` exists (prompt 16) and
    that its source carries a ``pytest.mark.e2e`` marker, preventing
    accidental deletion or demotion of the reproducibility harness.
  - ``backtest_spec_present`` — confirms
    ``docs/specs/backtest_spec.md`` (prompt 11) is still on disk.
  - ``red_team_matrix_present`` — confirms
    ``tests/adversarial/labels.json`` (prompt 13) is present, parses
    as a JSON list, and that every row carries an
    ``expected_verdict`` field — catching silent truncation of the
    adversarial corpus.
  - ``admin_dashboard_router_registered`` — imports
    ``server/fund/routers/admin_ui.py`` (prompt 19), asserts its
    router prefix is ``/admin``, and statically confirms
    ``server/fund/app.py`` both imports and mounts
    ``admin_ui_router``.
  - ``status_doc_prompt_recap`` and
    ``continuation_doc_prompt_recap`` — scan
    ``COHERENCE_ENGINE_PROJECT_STATUS.txt`` and

    tracking documents can never silently drift out of sync with the
    prompt ledger.

Exit semantics are deliberately coarse so CI steps can branch cheaply:
``0`` means every check passed, ``1`` means one or more checks
reported a soft failure (missing file, missing recap line, unsigned
marker), and ``2`` is reserved for loader / fixture errors where a
check itself raised an exception before it could reach a pass/fail
verdict. JSON output is byte-stable — keys are sorted, the check
registry is a tuple, and timestamps are excluded from the report
body — so a ``diff`` between two runs on an unchanged tree is empty.
A Markdown summary is emitted alongside the JSON for human review.

**New: Make target and runbook.**

``make release-readiness`` runs the script with
``PYTHONPATH=$(REPO_PARENT)`` (to match how the fund app is executed
elsewhere in the Makefile) and pipes outputs to
``artifacts/release-readiness.json`` and
``artifacts/release-readiness.md``. ``docs/ops/release_readiness.md``
documents the quick invocation, the exit-code contract, the report
shape, and a per-check failure-mode table so operators can diagnose a
red checklist without having to read the script.

**New: test suite.**

``tests/test_release_readiness_check.py`` loads the script as a module
via ``importlib.util`` (the script is not a package member; this
mirrors the approach already used by other ``deploy/scripts/`` tests)
and covers:

  - the happy path: ``main()`` exits ``0`` on the current tree and
    every registered check reports ``pass`` with no failures or
    loader errors;
  - report structure guarantees: rows follow the ``CHECKS`` tuple
    order, all ``check_id`` values are unique and non-empty, and the
    Markdown summary mentions every ``check_id``;
  - byte-stability: running the script twice on the same tree
    produces identical JSON (guarding determinism in CI);
  - ``_canonical_json`` sorts keys and indents by two spaces;
  - three forced-failure scenarios that ``shutil.move`` a required
    file (``tests/adversarial/labels.json``,
    ``docs/specs/backtest_spec.md``,
    ``tests/integration/test_e2e_pipeline.py``) aside within a
    ``try/finally`` block, assert exit code ``1`` and the specific
    ``reason_code`` (``red_team_matrix_missing``,
    ``backtest_spec_missing``, ``e2e_test_missing``), and restore the
    file regardless of assertion outcome so local developer trees are
    not corrupted;
  - a subprocess test that runs the script end-to-end via
    ``python3 deploy/scripts/release_readiness_check.py`` to prove
    the CLI entrypoint used by the Make target stays wired up.

**Docs sync.**

``COHERENCE_ENGINE_PROJECT_STATUS.txt`` and
``COHERENCE_ENGINE_CONTINUATION_PROMPT.txt`` now both open with a

**Explicit non-scope.**

no new event types, no new migrations, no new provider integrations.
