# Twelve-factor configuration audit

> **Code:** `server/fund/config.py`, `server/fund/services/env_gates.py`,
> `deploy/scripts/audit_twelve_factor.py`.
> **Tests:** `tests/test_twelve_factor_audit.py`.
> **Report:** `data/governed/config_audit_report.json` (regenerated by
> `python -m coherence_engine config audit --output ...`).

The Coherence fund backend follows the [12-factor](https://12factor.net)
config rules: **all configuration comes from the environment, every read
goes through one place, and behavior differences across environments are
declared explicitly.** This spec records the rules, the enforcement
machinery, and the remediation backlog.

## Environment taxonomy

The runtime is one of exactly four environments, resolved once at
startup from `COHERENCE_FUND_ENV` (preferred) or `APP_ENV`:

| token     | description                                              |
|-----------|----------------------------------------------------------|
| `dev`     | local laptop, ephemeral data, dry-run backends OK.       |
| `test`    | unit / integration / CI; isolated DB; mocks allowed.     |
| `staging` | pre-prod; real backends; same shape as `prod`.           |
| `prod`    | customer-facing; strict invariants; no debug surface.    |

Common spellings (`production`, `development`, `local`, `stage`, `qa`,
...) are normalized to one of the four canonical tokens by
`server/fund/config.py::_resolve_environment_token`. Any other value
raises at construction time — silent fallbacks are forbidden.

`environment` is **never** auto-detected from hostname, cloud metadata,
or any other implicit signal. It is always an explicit env var.

## The single config layer

`server/fund/config.py` holds the only `os.environ` reads in the
runtime app code. It exposes a Pydantic v2 `BaseSettings` model
(`FundSettings`) with one module-level instance (`settings`) that every
other module imports.

Backwards-compatible UPPER_CASE attribute aliases (`settings.DATABASE_URL`,
`settings.WORKER_BACKEND`, ...) are preserved as `@property` accessors
on the model so the migration to Pydantic does not require a sweep of
every call site in the same change.

Cross-field invariants live in a `model_validator(mode="after")` and
are enforced strictly when `environment == "prod"`:

* `STORAGE_BACKEND=local` rejected.
* `COHERENCE_FUND_AUTO_CREATE_TABLES=true` rejected (Alembic owns
  the schema in prod).
* `sqlite:///` database URLs rejected.
* `COHERENCE_FUND_SECRET_MANAGER_PROVIDER=disabled` rejected.

These are the values that *cannot* appear in prod regardless of
operator intent; loosening any of them requires editing the model
and the spec — not flipping an env var.

## The gate pattern

Behavior that legitimately differs between environments is declared in
exactly one place: `server/fund/services/env_gates.py`. Each gate is a
pure read of `settings.environment` plus a docstring explaining *why*
the split exists.

```python
from coherence_engine.server.fund.services.env_gates import (
    is_prod, allow_dry_run_backends, allow_debug_routes,
)

if allow_debug_routes():
    app.include_router(debug_router, prefix="/debug")
```

**Rules of thumb:**

* If a value differs by env, it is a `Settings` field, not a
  per-environment hardcoded constant.
* If *behavior* differs by env, it is a gate function in `env_gates.py`,
  not an inline `if env == "prod"` check.
* New gates must document **why** the split exists. Drift starts when
  someone copies a check without thinking about whether it should
  have been a Settings field instead.

## Auditor rules

`deploy/scripts/audit_twelve_factor.py` walks the codebase with `ast`
and reports on these rule families:

| rule                       | severity                               | notes                                                                 |
|----------------------------|----------------------------------------|-----------------------------------------------------------------------|
| `env_read_outside_config`  | error in `server/fund/`, warn in tooling | `os.environ.get` / `os.getenv` / `os.environ[...]` outside the allowlist. |
| `hardcoded_url`            | warn                                   | `https?://host` literals. `*.local`, `*.internal`, `localhost` allowed. |
| `print_in_runtime_code`    | warn                                   | `print(...)` outside `cli.py` / `deploy/scripts/` / `tests/`.         |
| `http_call_without_timeout`| warn                                   | `requests.{verb}` / `httpx.{verb}` calls missing the `timeout=` kwarg. |

The auditor's allowlist for env reads is intentionally tight:

* `server/fund/config.py` — the config layer itself.
* `server/fund/services/secret_manager.py`,
  `server/fund/services/secret_backends.py`,
  `server/fund/services/secret_manifest.py` — the secrets layer
  needs to read backend-selection env vars before it can route
  through Settings.
* `deploy/scripts/secret_manager_preflight.py`,
  `deploy/scripts/audit_twelve_factor.py` — operational tooling.
* `alembic/env.py` — alembic config bootstrap.

`deploy/scripts/` and `cli.py` are downgraded to **warn** (operator
flags like `CONFIRM_PRINT_SECRET=YES` legitimately come from the
environment), but every read still appears in the report so it can
be reviewed.

## Running the audit

```sh
# Full JSON report:
python -m coherence_engine config audit --output data/governed/config_audit_report.json

# Quick summary (exit 2 if any errors):
python -m coherence_engine config audit --format summary

# Show resolved Settings (with SecretStr fields redacted):
python -m coherence_engine config show
```

CI should run `python -m coherence_engine config audit` and fail the
build on a non-zero exit.

## Remediation backlog

The first run of the auditor surfaces the legacy violations introduced
before this spec landed. They are recorded verbatim in
`data/governed/config_audit_report.json` and intentionally not
silenced. Future Wave 9 prompts route each violation either to a
`Settings` field (preferred) or to the `env_gates` module, depending
on whether the underlying behavior is a value-split or a
behavior-split.

The audit JSON is the source of truth for the backlog — work it down
top-of-list.

## Prohibitions (the rule shape, restated)

* Do not `os.environ.get` / `os.getenv` outside the allowlist.
* Do not introduce per-environment hardcoded constants. If a value
  differs by env, it is a Settings field.
* Do not auto-flip `environment` based on hostname or any implicit
  signal.
* Do not use `print` for runtime logging. Use `logging` with a
  module-level logger.
* Do not call `requests.get(url)` without a `timeout=`.
